# 判断触发启发规则

> 指导 AI 在代码生成过程中判断"哪些决策需要记录"的规则。
> 与 judgment-log-format.md 配合使用——格式规范定义"怎么写"，本文件定义"什么时候写"。

## 核心原则

**不是所有选择都是判断，也不是所有判断都需要人确认。**

- 编码标准有明确规定的 → 不记录（直接执行）
- 行为契约有明确约束的 → 不记录（直接执行）
- 行为契约有歧义或 standards 未覆盖的 → 记录为"需要确认"
- 行为契约未提及但 AI 必须做选择的 → 记录为"已做判断"

## 决策树

当 AI 在代码生成过程中遇到一个选择点时，按以下顺序判断：

```
1. 编码标准(standards.md)是否有明确规定？
   → 是：直接执行，不记录
   → 否：继续 ↓

2. 代码模式(patterns/)是否有标准做法？
   → 是：按模式执行，记录为"已做判断"（引用 patterns）
   → 否：继续 ↓

3. 行为契约(contract)是否有明确约束？
   → 是：按约束执行，不记录
   → 约束有歧义/多种解读：记录为"需要确认"
   → 否：继续 ↓

4. 项目已有代码（参考实现）是否有先例？
   → 是：按先例执行，记录为"已做判断"（引用参考文件）
   → 否：记录为"需要确认"
```

## "需要确认"的触发条件

以下情况**必须**标记为 ⚠️ 需要确认：

### 行为契约歧义

| 触发条件 | 示例 |
|---------|------|
| 同一规则有两种以上合理解读 | BR-001 写"超时30分钟"，未指定超时检查机制（轮询？延迟队列？定时任务？） |
| 规则之间存在矛盾 | BR-002 要求"实时处理"，BR-005 要求"批量执行" |
| 行为契约未覆盖的边界条件 | 状态机定义了正常流程，但未定义并发场景下的处理 |
| 验收条件无法直接验证 | "性能良好"不可量化，需要确认具体指标 |

### Standards 未覆盖的模式选择

| 触发条件 | 示例 |
|---------|------|
| 幂等策略选择 | 数据库唯一索引 vs 分布式锁 vs Token 机制 |
| 并发控制策略 | 乐观锁 vs 悲观锁 vs 分布式锁 |
| 缓存策略 | 是否需要缓存、缓存失效策略 |
| 消息可靠投递 | 同步调用 vs 异步消息 vs 事件驱动 |
| 查询优化 | 单次大查询 vs 分页多次查询 vs 批量加载 |

### 与项目已有代码不一致

| 触发条件 | 示例 |
|---------|------|
| 同 domain 已有不同实现风格 | 项目中 Entity 有充血模型和贫血模型两种风格 |
| 新增接口但项目有命名先例不一致 | 项目中既有 XxxManageFacade 也有 XxxService 暴露接口 |
| 需要引入行为契约未提及的外部依赖 | 需要新增 Redis/MQ/定时任务框架 |

### 性能与兼容性权衡

| 触发条件 | 示例 |
|---------|------|
| 规则实现方式影响性能 | BR-001 超时机制：轮询有延迟但简单，延迟队列精确但复杂 |
| 兼容性约束与最佳实践冲突 | 历史接口命名不规范，新代码是否遵循旧命名 |
| 数据量预估影响技术选择 | 不确定数据量级，选择分页策略还是流式处理 |

## "已做判断"的触发条件

以下情况记录为已做判断（人可快速扫过）：

| 触发条件 | 记录格式 |
|---------|---------|
| patterns 中有标准模式，AI 选择套用 | `参考: standards/patterns/{pattern}.md` |
| standards 有明确约束，AI 按约束选择 | `参考: standards/{section}` |
| 行为契约有明确规则，AI 选择一种实现方式 | `参考: contract BR-xxx` |
| 项目已有代码有先例，AI 遵循先例 | `参考: {source-file}:{line}` |
| 多种技术方案等价，AI 选择更简单的 | `原因: {为什么等价}，选择 {chosen} 因为 {reason}` |

## 不需要记录的决策

以下情况**不记录**到判断日志：

| 类别 | 示例 | 原因 |
|------|------|------|
| standards 明确规定的 | Entity 用 @Getter/@Setter 不用 @Data | 编码标准已约定，无需人再确认 |
| 项目统一约定的 | 包命名规则、DO 注解风格 | 遵循即可，不是判断 |
| 唯一选择的 | 行为契约只定义了一种状态转换路径 | 没有选择余地 |
| 纯机械的代码生成 | 根据 API 契约生成 Command/VO 字段 | 契约已精确定义 |
| 命名按照规范 | 类名、方法名遵循命名规范 table | 规范已覆盖 |

## 判断密度指南

不同复杂度的 Feature，判断日志的预期密度：

| Feature 复杂度 | 领域模型数 | 预期已做判断 | 预期需要确认 |
|---------------|-----------|-------------|-------------|
| 简单 CRUD | 1 个 Entity | 2-5 条 | 0-1 条 |
| 标准业务流程 | 2-3 个 Entity + 状态机 | 5-10 条 | 1-3 条 |
| 复杂业务逻辑 | 3+ Entity + 策略 + 外部集成 | 10-20 条 | 3-5 条 |

**如果判断日志远少于预期**：可能 AI 没有主动暴露决策，需要检查。
**如果判断日志远多于预期**：可能行为契约约束不够精确，建议回到 clarify 补充约束。

## 判断升级：从"已做判断"到"需要确认"

在实现过程中，以下情况应将已做判断升级为需要确认：

1. **实现时发现行为契约规则有隐含歧义**：最初认为规则明确，实现中发现两种解读
2. **参考的 patterns/standards 相互矛盾**：两个标准给出不同建议
3. **性能测试结果与预期不符**：选择在理论上合理，但实际表现不达标
4. **与现有代码产生冲突**：按先例执行后发现与新需求矛盾

升级格式：
```
- [JD-xxx] ⚠️ {原决策描述}（升级自 JD-yyy）
  原因: {实现中发现的新信息}
  选项: A: {...}, B: {...}
  建议: {AI 的推荐}
```
